##  Вопросы для самопроверки

**1. Почему мы создаем Deployment, а не сразу Pod?**  
Pod — это минимальная единица в Kubernetes, но он эфемерен: при сбое, обновлении или перезапуске ноды под удаляется и создаётся заново с другим именем и IP. Если создать под напрямую (`kubectl run`), Kubernetes не будет его мониторить и перезапускать при падении. Deployment — это контроллер более высокого уровня, который управляет жизненным циклом подов: он гарантирует желаемое количество реплик (`replicas`), автоматически перезапускает упавшие поды, поддерживает rolling updates и откаты. То есть Deployment добавляет декларативность, отказоустойчивость и управляемость, которых нет при ручном создании подов.

**2. В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?**  
Liveness Probe отвечает на вопрос «Жив ли контейнер?». Если проверка падает несколько раз подряд (`failureThreshold`), Kubernetes помечает контейнер как мёртвый и **перезапускает** его (лечит зависшие приложения, дедлоки, утечки памяти). Readiness Probe отвечает на вопрос «Готов ли контейнер принимать трафик?». Если она падает, под **не перезапускается**, но временно исключается из балансировки нагрузки сервиса (не получает новые запросы). Это критично во время старта приложения, миграций БД или загрузки кэша: под остаётся живым, но не получает трафик, пока не станет полностью готовым. Разница: liveness → перезапуск, readiness → исключение из Service.

**3. Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?**  
`requests` — это гарантированный минимум ресурсов (CPU в миллиядрах, память в мибибайтах), который Kubernetes резервирует для пода при планировании на ноду. Планировщик не разместит под на ноде, если там не хватает запрошенных ресурсов. `limits` — это максимум, который контейнер может использовать. Если контейнер превысит `limits.memory`, он будет убит по событию OOMKilled. Если превысит `limits.cpu`, процесс будет троттлиться (замедлен), но не убит. Указывать их важно по трём причинам: (1) предотвращает «шумных соседей» от захвата всех ресурсов ноды; (2) помогает K8s эффективно упаковывать поды по кластеру (bin packing); (3) делает поведение приложения предсказуемым при нагрузке и упрощает отладку проблем с производительностью.

**4. Как Service находит свои Pod-ы? (Подсказка: Labels & Selectors).**  
Service использует механизм `selector` — набор меток (labels), например `app: shipments-app`. В Deployment'е каждый под получает эти метки через `spec.template.metadata.labels`. Kubernetes автоматически отслеживает все поды с совпадающими метками и добавляет их текущие IP-адреса в эндпоинты сервиса (`Endpoints` или `EndpointSlice` объект). Когда ты делаешь запрос на `shipments-svc-s16:8230`, kube-proxy (или CNI-плагин в режиме IPVS/iptables) перенаправляет трафик на один из здоровых подов (обычно round-robin). Если под умирает и перезапускается с новым IP, Service автоматически обновляет список эндпоинтов — приложение не требует переконфигурации или перезапуска. Это обеспечивает стабильный адрес для динамически меняющегося набора подов.

**5. Какой тип сервиса (ClusterIP, NodePort, LoadBalancer) используется для внутреннего общения, а какой — для внешнего доступа?**  
- **ClusterIP** (дефолт): сервис получает виртуальный IP, доступный **только внутри кластера**. Идеален для внутреннего общения между микросервисами (например, фронтенд → бэкенд, бэкенд → база).
- **NodePort**: открывает статический порт на **каждой ноде** кластера (например, 30080). Трафик на `NODE_IP:30080` перенаправляется в сервис. Удобно для тестового внешнего доступа без облачного провайдера.
- **LoadBalancer**: создаёт внешний балансировщик через облачного провайдера (AWS ELB, GCP LB, Azure LB) и выделяет публичный IP. Трафик идёт `Публичный-IP:порт → NodePort → ClusterIP → Pod`. Используется для продакшен-доступа из интернета.

Для внутреннего общения почти всегда используют ClusterIP. Для внешнего — NodePort (dev/stage) или LoadBalancer (prod), либо Ingress-контроллер для HTTP/HTTPS маршрутизации.

**6. Что такое "Rolling Update" в контексте Деплоймента?**  
Rolling Update — это стратегия обновления приложения без простоя. При изменении образа или конфигурации в Deployment:
1. Kubernetes создаёт новый ReplicaSet с обновлённой версией.
2. Постепенно заменяет старые поды на новые: сначала создаёт один-два новых пода, ждёт, пока они пройдут Readiness Probe.
3. Только после этого удаляет соответствующее количество старых подов.
4. Процесс повторяется, пока все поды не будут обновлены.

Параметры `strategy.rollingUpdate.maxSurge` (сколько лишних подов можно создать сверх желаемого) и `maxUnavailable` (сколько подов можно временно убрать) позволяют контролировать скорость и доступность. Если что-то пошло не так, можно выполнить `kubectl rollout undo deployment/<name>` для мгновенного отката. Это обеспечивает нулевой или минимальный простой при деплое новых версий.
