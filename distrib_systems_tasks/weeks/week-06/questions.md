# Вопросы для самопроверки

## 1. Зачем нужны `variables` в GraphQL запросе? Почему нельзя просто склеивать строки?
1. Разделение логики — отделяем запрос от данных

2. Переиспользование — Один запрос можно использовать с разными параметрами

3. Кэширование — клиенты (Apollo) кэшируют запрос без переменных, а переменные подставляются отдельно

4. Избегание экранирования — не нужно экранировать кавычки и спецсимволы

## 2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?

**query** (обязательное) — GraphQL запрос или мутация
**variables** (опциональное) — объект с переменными
**operationName** (опциональное) — имя операции (нужно, если в запросе несколько операций)

## 3. Почему у GraphQL обычно только один URL endpoint?
1. Единая точка входа — вся логика API сосредоточена в одном месте, маршрутизация происходит на уровне полей запроса, а не URL

2. Гибкость — клиент сам решает, какие данные получить, а не сервер

3. Упрощение инфраструктуры — не нужны reverse-proxy правила для разных эндпоинтов

4. Само-документируемость — схема доступна через интроспекцию

5. Версионирование — не нужно версионировать URL, изменения в схеме обратно-совместимы

## 4. Что означает наличие поля `errors` в ответе сервера? Может ли оно прийти вместе с `data`?

Поле errors означает, что при выполнении запроса произошли ошибки (валидация, резолверы, аутентификация и т.д.).

Да, errors и data могут приходить одновременно.

- Если ошибка в одном поле, остальные поля возвращаются

- data содержит успешно загруженные данные

- errors описывает, что не удалось загрузить

## 5. Как клиент должен обрабатывать ошибки, если HTTP статус ответа 200?

GraphQL всегда возвращает HTTP 200, даже при ошибках выполнения. Клиент должен:
1. Всегда проверять поле errors в теле ответа
2. Обрабатывать частичные данные (data может быть null или содержать null значения)

## 6. В чем плюсы использования типизированных клиентов (например, Apollo Client или Relay) перед простыми HTTP-запросами?

## В чем плюсы использования типизированных клиентов (Apollo Client, Relay) перед простыми HTTP-запросами?

| Возможность | Простой `requests` | Apollo Client / Relay |
|-------------|-------------------|----------------------|
| **Типизация** | ❌ Ручная или через `TypedDict` | ✅ Автогенерация типов из схемы + TypeScript |
| **Валидация запросов** | ❌ Только на сервере | ✅ На этапе сборки/компиляции |
| **Кэширование** | ❌ Ручная реализация | ✅ Встроенное нормализованное кэширование |
| **Оптимистичные обновления** | ❌ Ручная логика | ✅ Встроенная поддержка |
| **Рефетчинг/пагинация** | ❌ Ручная реализация | ✅ Встроенные хуки (`useQuery`, `useMutation`) |
| **Автодополнение в IDE** | ❌ Только строки | ✅ Автокомплит полей запроса |
| **Управление состоянием** | ❌ Отдельно (Redux, Zustand) | ✅ Встроено в клиент |
| **Обработка загрузки/ошибок** | ❌ Ручная | ✅ Автоматическая через состояния |
| **Подписки (WebSocket)** | ❌ Ручная реализация | ✅ Встроенная поддержка |
| **Обновление UI после мутаций** | ❌ Ручное | ✅ Автоматическое через кэш |
| **Типобезопасность** | ❌ Ошибки в runtime | ✅ Ошибки на этапе сборки |
