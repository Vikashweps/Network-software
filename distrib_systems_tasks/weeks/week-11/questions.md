##  Вопросы для самопроверки

**1. Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?**  
По умолчанию Docker Compose создаёт общую bridge-сеть для всех сервисов в файле, и этого достаточно для базового взаимодействия. Однако пользовательская сеть даёт больше контроля: можно изолировать группы сервисов друг от друга (например, отделить бэкенд от фронтенда), задать статические IP-адреса для контейнеров, настроить DNS-разрешение по коротким именам без суффиксов проекта. Кроме того, в пользовательской сети контейнеры автоматически резолвят имена сервисов (например, `redis` → IP), тогда как в дефолтной сети иногда требуются полные имена (`project_redis_1`). Для продакшена это критично: явная декларация сетей делает архитектуру прозрачнее и безопаснее.

**2. В чем разница между `build` и `image` в описании сервиса?**  
`image` указывает на готовый образ в реестре (Docker Hub, GCR и т.д.), который нужно просто скачать и запустить. Это удобно для стандартных сервисов вроде `redis:7-alpine` или `nginx:alpine`. `build` говорит Compose, что образ нужно собрать локально из `Dockerfile`: указывается путь к контексту (`context: ./app`) и имя файла (`dockerfile: Dockerfile`). Это необходимо для твоих сервисов, код которых ты разрабатываешь. Можно комбинировать: `build` для своих приложений, `image` для сторонних зависимостей. При повторной сборке Docker кэширует слои, поэтому `build` не всегда означает долгую компиляцию.

**3. Что делает `depends_on`? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?**  
Обычный `depends_on: [redis]` лишь гарантирует порядок запуска: Docker сначала создаст и запустит контейнер `redis`, и только потом — зависимый сервис. Но это не ждёт, пока приложение внутри `redis` реально будет готово принимать соединения (например, закончит загрузку данных или пройдёт миграции). Поэтому приложение может стартовать, попытаться подключиться к ещё инициализирующейся базе и упасть с ошибкой. Чтобы решить это, используют `depends_on` с условием `condition: service_healthy` вместе с `healthcheck`: Compose ждёт, пока проверка здоровья зависимого сервиса начнёт возвращать успех, и только тогда запускает следующий. Это надёжнее, но требует корректно настроенных healthcheck-ов.

**4. Как работают healthcheck-и и как использовать их в `depends_on`?**  
Healthcheck — это периодическая команда, которая проверяет, что сервис внутри контейнера жив и готов обрабатывать запросы. Настраивается через `test` (команда), `interval` (период), `timeout` (время ожидания), `retries` (число попыток до пометки `unhealthy`) и `start_period` (время на старт перед проверками). Пример для FastAPI: `python -c "import urllib.request; urllib.request.urlopen('http://localhost:8107/health')"`. Чтобы использовать в `depends_on`, указывают: `depends_on: { redis: { condition: service_healthy } }`. Тогда Compose не запустит зависимый сервис, пока `redis` не станет `healthy`. Это критично для надёжности: приложение не стартует, пока база не готова, гейт не шлёт трафик на мёртвый бэкенд.

**5. Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?**  
**Способ 1: прямо в `docker-compose.yml`** через секцию `environment`:
```bash
environment:
  - REDIS_URL=redis://redis:6379/0
  - DEBUG=false
```
**Способ 2:** через файл .env в корне проекта::
```bash
# .env
REDIS_URL=redis://redis:6379/0
DEBUG=false
```

Compose автоматически подгрузит переменные из .env. Также можно использовать env_file: .env для загрузки всего файла. Переменные из .env удобно не коммитить в репозиторий (добавить в .gitignore), а передавать секреты через docker secret или внешние менеджеры.

**6. Почему Docker Compose редко используют в продакшене (хотя это возможно)?**
Docker Compose отлично подходит для локальной разработки, тестирования и небольших проектов, но в продакшене часто используют оркестраторы уровня Kubernetes, Nomad или Swarm. Причины:

- Масштабирование: Compose не умеет автоматически масштабировать сервисы по нагрузке, тогда как K8s делает это «из коробки».

- Отказоустойчивость: при падении хоста Compose не перезапустит контейнеры на другой ноде, а оркестратор — сделает.

- Обновления без простоя: rolling updates, canary-деплои, откаты — в Compose это ручная работа, в K8s — декларативная.

- Безопасность: управление секретами, RBAC, сетевые политики в Compose ограничены.

- Мониторинг и логи: интеграция с Prometheus, Loki, Jaeger проще через стандарты оркестраторов.


# Запуск 
docker-compose up --build -d

#  Проверка статуса сервисов
docker-compose ps
# Ожидем: все сервисы → (healthy)

# Тест через Gateway (порт 80)
curl http://localhost/products/
curl -X POST http://localhost/products/ \
  -H "Content-Type: application/json" \
  -d '{"name":"Тест","description":"Описание","price":99.0,"stock_quantity":10}'

#  Тест напрямую (порт 8107)
curl http://localhost:8107/health

#  Логи (отладка)
docker-compose logs -f          # все сервисы
docker-compose logs sessions-svc -f  # только бэк

#  Остановка
docker-compose down             # сохранить данные
docker-compose down -v          # удалить volumes (полная очистка)
