# Вопросы для самопроверки

## 1. В чем главное концептуальное отличие GraphQL от REST?

| Характеристика | REST | GraphQL |
|----------------|------|---------|
| **Эндпоинты** | Множественные (/users, /posts, /comments) | Единый (/graphql) |
| **Структура ответа** | Сервер определяет | Клиент определяет |
| **Over-fetching** | Есть (получаем лишние данные) | Нет |
| **Under-fetching** | Есть (нужно несколько запросов) | Нет |
| **Связанные данные** | Нужно несколько запросов | Один запрос |

**Главное отличие:** В REST клиент получает то, что сервер решил отдать. В GraphQL клиент получает точно то, что запросил.

## 2. Что такое Schema Definition Language (SDL)?
SDL - язык описания схемы GraphQL
```bash
type Product {
  id: ID!
  name: String!
  price: Float!
  inStock: Boolean!
}

type Query {
  products: [Product!]!
  product(id: ID!): Product
}

type Mutation {
  createProduct(input: ProductInput!): Product!
}

input ProductInput {
  name: String!
  price: Float!
  inStock: Boolean = true
}
```
**Зачем нужен SDL:**
- Контракт между клиентом и сервером
- Валидация запросов на стороне сервера
- Автодокументация и автодополнение в IDE (GraphiQL)
- Генерация типов для клиентских приложений

## 3. Чем Query отличается от Mutation? Могут ли они выполняться параллельно?

| Характеристика | Query | Mutation |
|----------------|-------|----------|
| **Назначение** | Чтение данных | Изменение данных |
| **Идемпотентность** | Да | Нет |
| **Выполнение** | Параллельно | Последовательно |
| **Семантика** | GET | POST/PUT/DELETE |

## 4. Что такое "резолвер" (resolver) и кто его вызывает?
Резолвер - функция, возвращающая данные для поля
**Кто вызывает:**
1. GraphQL-сервер при обработке запроса
2. Для каждого поля в запросе находит соответствующий резолвер
3. Вызывает резолверы в иерархическом порядке
4. Собирает результат

**Структура вызова:**
Запрос → Парсинг → Вызов корневого резолвера → Вызов дочерних резолверов → Ответ

## 5. Как решается проблема N+1 запросов в GraphQL (dataloader)?

**Проблема N+1 запросов** возникает в GraphQL, когда запрашиваются связанные данные. Например, при запросе списка продуктов и их категорий

Что происходит без оптимизации:

1. Делается 1 запрос к БД для получения всех продуктов
2. Для каждого из 100 продуктов делается отдельный запрос для получения категории
3. Итого: 101 запрос (1 + 100)

Решение — DataLoader:

**DataLoader** — это утилита, которая решает проблему N+1 через два механизма:

1. Batching (группировка): собирает все запросы за один tick event loop и выполняет их одним пакетным запросом к БД
2. Caching (кэширование): в рамках одного HTTP-запроса повторные запросы к одному объекту не идут в БД

## 6. Почему в GraphQL сложно реализовать кэширование на уровне HTTP (как в REST)?

1. Один URL для всех запросов — все запросы идут на /graphql, а не на разные эндпоинты как в REST

2. Разные запросы выглядят одинаково — запросы { user { name } } и { user { name email } } идут на один URL, но возвращают разные данные, и HTTP кэш не может их различить

3. POST вместо GET — GraphQL использует POST (не кэшируется), а REST для чтения использует GET (кэшируется)

**Решение:** кэширование делают на клиенте (Apollo Client), а не на уровне HTTP.
