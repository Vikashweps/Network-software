##  Вопросы для самопроверки

**1. В чем разница между Continuous Integration, Continuous Delivery и Continuous Deployment?**  
- **Continuous Integration (CI)** — практика частого слияния кода разработчиков в общую ветку с автоматической сборкой и запуском тестов. Цель: быстро находить конфликты и баги.  
- **Continuous Delivery (CD)** — автоматическая подготовка релиза к развёртыванию: код проходит все тесты, собирается в артефакт и может быть задеплоен вручную одной командой. Деплой в прод — решение человека.  
- **Continuous Deployment (CD)** — полностью автоматический деплой в продакшен после прохождения всех проверок. Никаких ручных действий: если тесты зелёные, код попадает к пользователям.  
Ключевое отличие: **CI** — интеграция и тестирование, **Delivery** — готовность к деплою (ручной триггер), **Deployment** — авто-деплой без участия человека.

**2. Почему тесты должны запускаться *до* сборки Docker образа (или хотя бы до его пуша)?**  
Сборка и публикация образа — ресурсоёмкие операции: они тратят время раннера, место в реестре и могут затриггерить последующие шаги (деплой). Если в коде есть ошибка, а образ уже собран и запушен, в реестр попадёт битый артефакт, который может случайно быть использован в проде. Запуск тестов на ранней стадии («fail fast») экономит ресурсы, даёт быструю обратную связь разработчику и гарантирует, что в реестр и дальше по пайплайну попадают только проверенные артефакты.

**3. Зачем нужно кэширование зависимостей в CI пайплайне?**  
Без кэша каждый запуск пайплайна заново скачивает пакеты (`pip install`, `npm install` и т.д.), что может занимать минуты или даже часы. Кэширование позволяет переиспользовать уже загруженные зависимости между запусками, ускоряя сборку в разы. В GitHub Actions это делается через `actions/setup-python@v5` с параметром `cache: 'pip'` или через `actions/cache@v3` для произвольных путей. Это особенно важно для больших проектов с сотнями зависимостей и для частых коммитов.

**4. Что такое "Secret" в контексте CI/CD? Как безопасно передать пароль от Docker Hub в пайплайн?**  
Secret (секрет) — это чувствительные данные (токены, пароли, ключи), которые нельзя хранить в открытом виде в репозитории. В платформах типа GitHub Actions или GitLab CI секреты шифруются, не логируются и передаются в раннер только как переменные окружения. Чтобы безопасно передать пароль от Docker Hub:
1. Создай Access Token в [Docker Hub Security](https://hub.docker.com/settings/security).
2. Добавь его в репозиторий: **Settings → Secrets and variables → Actions** как `DOCKERHUB_TOKEN`.
3. Используй в пайплайне: `password: ${{ secrets.DOCKERHUB_TOKEN }}`.
Никогда не хардкодь секреты в код или файл пайплайна — они могут утечь в логи или историю коммитов.

**5. Что такое "Matrix Build" (матричная сборка)?**  
Matrix Build — это возможность запустить один и тот же джоб пайплайна с разными параметрами параллельно. Например, протестировать код на нескольких версиях Python (`3.9`, `3.10`, `3.11`) или на разных ОС (`ubuntu-latest`, `windows-latest`, `macos-latest`). В GitHub Actions это настраивается через `strategy.matrix`:
```yaml
strategy:
  matrix:
    python-version: ["3.9", "3.10", "3.11"]
    os: [ubuntu-latest, windows-latest]

```

Это позволяет убедиться, что приложение работает в разных окружениях, не дублируя код пайплайна.

**6. Почему нельзя хранить конфиги продуктовой среды в репозитории кода (или можно, но осторожно)?**

Продуктовые конфиги часто содержат секреты (пароли БД, API-ключи), специфичные для продакшена настройки (масштабирование, лимиты) или чувствительные данные (домены, IP). Их хранение в открытом репозитории создаёт риски утечки, несанкционированного доступа и случайного применения dev-конфигов в проде.
Можно хранить, если:

   - Конфиг не содержит секретов (используются переменные окружения или внешние менеджеры секретов).
   - Используется шаблонизация (Helm, Kustomize) с разделением values-prod.yaml в приватном репозитории.
   - Включены проверки доступа (branch protection, code owners) и аудит изменений.
