##  Вопросы для самопроверки

**1. Что такое Server Streaming? Пример?**
Server Streaming — это когда клиент отправляет один запрос, а сервер в ответ присылает не один готовый результат, а поток данных, которые приходят по мере готовности. Это удобно, когда данных много или они генерируются в реальном времени. Примеры: лента новостей в соцсети, биржевые котировки, поток логов с сервера или загрузка большого файла частями.

> *Пример:* лента новостей, биржевые котировки, логи — данные приходят частями по мере появления.

**2. Чем стриминг отличается от получения большого списка (List)?**
| List (Unary) | Streaming |
|-------------|-----------|
| Ждём, пока сервер соберёт всё | Первый элемент приходит сразу |
| Сервер тратит много памяти на список | Сервер шлёт по одному, экономя память |
| Клиент получает «всё или ничего» | Можно остановить поток в любой момент |

При обычном запросе списка сервер сначала собирает все данные, потом отправляет их одним большим пакетом — клиент ждёт, пока всё подготовится. При стриминге сервер начинает отправлять данные сразу, по одному элементу. Это снижает задержку до первого байта, экономит память на сервере (не нужно хранить весь список перед отправкой) и позволяет клиенту начинать обработку раньше. Кроме того, клиент может в любой момент остановить поток, если ему хватило данных.

**3. Основные метрики производительности?**
- **Latency** — задержка (время ответа на 1 запрос, мс)
- **Throughput (RPS)** — пропускная способность (запросов в секунду)
- **CPU/Memory Usage** — загрузка процессора и памяти

**4. Зачем переиспользовать gRPC Channel при тестах?**
Создание канала — это установка TCP-соединения + настройка HTTP/2 (дорого).
Мы замеряем **скорость работы RPC**, а не скорость соединения. В реальных приложениях канал создают один раз и используют долго.

**5. Когда REST удобнее gRPC?**

REST удобнее, когда нужно сделать публичный API: в браузер или мобильное приложение не нужно добавлять генерацию кода из .proto-файлов, можно просто отправить HTTP-запрос. Также REST проще для быстрых прототипов и отладки — запрос можно посмотреть в браузере или Postman, не поднимая gRPC-клиент. Если команда не знакома с gRPC, REST позволит быстрее начать работу. И наконец, для очень маленьких и редких запросов накладные расходы на настройку gRPC могут «съесть» его преимущество в скорости.

**6. Что такое Head-of-Line Blocking и как HTTP/2 это решает?**
- **Проблема (HTTP/1.1):** В одном соединении запросы идут строго по очереди. Если первый «завис» — остальные ждут.
- **Решение (HTTP/2 / gRPC):** **Мультиплексирование** — множество запросов летят параллельно в одном соединении, не блокируя друг друга.
