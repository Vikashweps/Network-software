# Latency vs Throughput Analysis: items-s16

## Конфигурация теста
- **Проект**: items-s16
- **Сервис**: items-svc-s16 (порт 8281)
- **Endpoint REST**: http://localhost:8281/api/items
- **gRPC сервис**: items.v1.ItemsService
- **Инструменты**: wrk (REST), ghz (gRPC)
- **Duration**: 30s
- **Ограничения контейнера**: 0.5 CPU, 256MB RAM

## Метрики
Измерялись следующие показатели:
- **Latency**: среднее время отклика, а также перцентили P50, P95, P99
- **Throughput**: количество успешных запросов в секунду (RPS)
- **Error rate**: процент неудачных запросов

## Результаты: REST API (wrk)

| Concurrency | Avg Latency (ms) | P99 Latency (ms) | Requests/sec | Error Rate |
|-------------|-----------------|------------------|--------------|------------|
| 10          | 8.2             | 24.5             | 852          | 0.0%       |
| 100         | 35.4            | 118.7            | 2134         | 0.1%       |
| 1000        | 182.6           | 648.3            | 2387         | 2.5%       |

## Результаты: gRPC (ghz)

| Concurrency | Avg Latency (ms) | P99 Latency (ms) | Requests/sec | Error Rate |
|-------------|-----------------|------------------|--------------|------------|
| 10          | 3.8             | 11.2             | 2156         | 0.0%       |
| 100         | 17.5            | 54.8             | 5834         | 0.0%       |
| 1000        | 95.3            | 382.1            | 7245         | 0.5%       |

## Анализ зависимости Latency от RPS
- При низкой нагрузке (до ~200 RPS) latency растёт линейно.
- После точки насыщения (~2400 RPS для REST, ~7200 RPS для gRPC) наблюдается экспоненциальный рост задержек.
- gRPC демонстрирует меньшую latency при одинаковой нагрузке благодаря:
  - бинарному формату Protocol Buffers (меньший размер полезной нагрузки);
  - мультиплексированию потоков через HTTP/2;
  - более эффективной сериализации/десериализации.

## Точка насыщения
1. **REST**: точка насыщения достигается при ~2400 RPS, после чего P99 latency превышает 600 мс.
2. **gRPC**: точка насыщения смещена до ~7200 RPS, что на 200% выше, чем у REST.

## Выводы
1. Сервис **items-s16** демонстрирует ожидаемое поведение под нагрузкой: рост concurrency приводит к увеличению latency и выходу на плато по throughput.
2. gRPC показывает лучшую производительность по всем метрикам:
   - В 2.5-3 раза выше пропускная способность (7245 vs 2387 RPS при concurrency=1000)
   - В 2 раза меньшая средняя задержка (95ms vs 182ms)
   - В 1.7 раза меньший P99 (382ms vs 648ms)
3. Для продакшена рекомендуется:
   - использовать gRPC для service-to-service вызовов;
   - оставить REST для публичного API;
   - настроить горизонтальное масштабирование при приближении к точке насыщения.
