##  Информация о слоях
Каждая инструкция в `Dockerfile` (`RUN`, `COPY`, `ENV`, `EXPOSE`, `CMD`) создает отдельный read-only слой. В моем мультистейдж-образе:
- **Этап `builder`**: базовый образ `+ WORKDIR + COPY requirements.txt + RUN pip install` → создает слои с зависимостями и кэшем компилятора.
- **Этап `production`**: базовый образ `+ WORKDIR + COPY --from=builder + COPY app.py schemas.py + ENV + EXPOSE + CMD` → добавляет слои с готовыми библиотеками, кодом приложения и метаданными запуска.
Итоговый образ содержит ~10-12 слоёв (включая базовый `python:3.11-slim`). Благодаря тому, что `requirements.txt` копируется до кода, слои с зависимостями кэшируются и не пересобираются при изменении исходного кода.

##  Вывод
Использование двух этапов сборки и правильного порядка инструкций сократило размер образа и сделало сборку предсказуемой за счет кэширования слоёв.
 Чтобы посмотреть слои вручную:  
 ```bash
 docker history pizza-s16:latest
```


##  Docker: Сборка и запуск

### Команды для работы с образом

```bash
# 1. Удалить старый контейнер (если есть)
docker rm -f pizza-test

# 2. Собрать образ из Dockerfile в текущей папке
# -t pizza-s16 — задаёт имя и тег образа
# . — контекст сборки (текущая директория)
docker build -t pizza-s16 .

# 3. Запустить контейнер
# -d — фоновый режим (detached)
# -p 8116:8116 — проброс порта: хост:контейнер
# --name — удобное имя контейнера
docker run -d -p 8116:8116 --name pizza-test pizza-s16

# 4. Проверить, что приложение работает
curl http://localhost:8116/products/

# 5. Посмотреть размер образа (для отчёта)
docker images pizza-s16

# 6. Остановить и удалить контейнер (когда закончил)
docker stop pizza-test && docker rm pizza-test
```
